# 项目文件夹即案卷：搭建 SaaS 合同纠纷教学工作台

> 本课程用于讲解 Claude Code 垂直行业工作台方法，不提供法律行业解决方案。案例目录是教学结构，不是律所标准案卷规范。

## 本课目标

用项目目录讲清楚 Claude Code 的第一个核心能力：把一次聊天变成一个可持续维护、可追踪、可复盘的行业工作台。

## 本课要讲的 Claude Code 功能

- 用文件夹组织长期项目上下文。
- 区分原始材料、过程稿、交付稿和复盘资产。
- 让 Claude Code 在项目边界内读取、修改和沉淀文件。

## 为什么适合垂直行业知识工作

律师、法务、咨询、财税、医疗等知识工作都有大量材料和版本。项目目录能把“材料从哪里来、过程稿在哪里、客户交付在哪里、下次复用什么”分开，避免所有内容混在一次对话里。

## 核心概念

项目即工作台、材料分层、过程可追踪、交付隔离、复盘沉淀

## 本案中的演示任务

使用 `templates/case-workbench/` 创建 SaaS 合同纠纷教学工作台，把模拟合同、补充协议、邮件、聊天记录、付款记录、系统登录记录放入对应目录。

## 推荐提示词

```text
请基于 verticals/lawyer/templates/case-workbench 的目录结构，说明这个 SaaS 合同纠纷教学项目中每个目录应该放什么、哪些内容不能直接放入客户交付区、哪些材料需要先脱敏。
```

## 预期输出

目录说明、材料放置规范、内部过程区与客户交付区边界、后续课程会使用的文件路径索引。

## 学员如何替换成自己的 SOP

要求学员把目录名改成自己机构已经使用的分类方式，例如“客户原始材料”“内部分析过程”“正式交付文件”“复盘模板”。如果机构已有案卷规则，应以机构规则为准。

## 课堂练习

让学员拿一个非敏感历史项目，画出它的材料流向：进入项目目录、进入分析过程、进入交付包、进入复盘资产。不要上传真实敏感材料，只写目录和文件类型。

## 检查点 / 常见误区

- 原始材料和客户交付稿是否混放。
- 是否把未脱敏材料放进演示目录。
- 是否只建目录、不说明每个目录的使用规则。
- 是否能从目录结构看出后续课程的工作流。

## 讲课提示

本课重点不是“律师案卷该怎么归档”，而是“Claude Code 项目为什么需要清晰的上下文容器”。

## 本课交付物

一个 SaaS 合同纠纷教学项目目录，以及一份目录使用说明。
