# Reviewer/Evaluator：质量控制、交付包与课程闭环

> 本课程用于讲解 Claude Code 垂直行业工作台方法，不提供法律行业解决方案。Reviewer/Evaluator 演示质量控制结构，不替代律师复核。

## 本课目标

学习如何把复核从生成流程中独立出来，并把一次课程案例沉淀为客户交付包、内部复盘和下一次可复用模板。

## 本课要讲的 Claude Code 功能

- Reviewer / Evaluator 的独立复核角色。
- 输出质量检查清单。
- 客户版本与内部过程版本隔离。
- 项目复盘与模板沉淀。

## 为什么适合垂直行业知识工作

垂直行业交付通常有合规、专业、客户沟通和证据链要求。生成输出只是中间步骤，真正可交付前必须经过独立复核。Reviewer/Evaluator 的课程价值是把“检查什么、谁检查、如何记录问题”显性化。

## 核心概念

事实来源、敏感信息、内部痕迹清理、质量门禁、客户交付包、复盘模板

## 本案中的演示任务

使用 `verticals/lawyer/skills/lawyer-risk-review/` 和 `verticals/lawyer/rules/` 作为课堂样例，检查案件摘要、材料索引、示例审查表、证据目录和客户汇报草稿。重点讲复核结构，而不是给出最终法律判断。

## 推荐提示词

```text
请把 lawyer-risk-review 和 legal-* rules 当作教学模板，设计一份 Reviewer/Evaluator 复核清单。请检查来源、材料缺口、敏感信息、内部痕迹、过度承诺和必须人工确认事项。不要输出最终法律意见。
```

## 预期输出

复核清单、问题记录表、必须人工确认事项、客户交付包目录、复盘沉淀清单。

## 学员如何替换成自己的 SOP

要求学员把复核清单替换成本机构真实的质量控制标准，例如合伙人复核项、客户沟通红线、敏感信息处理规则、正式交付审批流程。

## 课堂练习

让学员拿前几课产出的一个草稿，设计一个独立 Reviewer 角色，只列问题、不重写正文。再把问题分成“必须修改”“需要人工确认”“可作为复盘模板优化”。

## 检查点 / 常见误区

- 是否让 Reviewer 继续生成正文。
- 是否没有区分客户版本和内部过程稿。
- 是否遗漏敏感信息和内部提示词痕迹。
- 是否没有把本次项目沉淀成下次可复用模板。

## 讲课提示

最后要回到课程定位：cc4Lawyer 交付的是一种搭建方法。法律团队需要用自己的 SOP、专业判断和复核标准完成真正的工作台。

## 本课交付物

Reviewer/Evaluator 复核清单、客户交付包目录、课程复盘与模板沉淀清单。
