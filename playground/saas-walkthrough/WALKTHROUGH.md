# WALKTHROUGH:SaaS 合同纠纷 Swarm 自动化执行指南

> **本文档是地图,不是车票**。真正"按一下就跑"的入口是本地 Skill:`/saas-case-swarm`。这份文档解释:Swarm 怎么组队、按什么节奏跑、在哪 4 个点会停下来问你、出问题怎么干预、为什么这样设计。

---

## 1. 你的角色升级了

| 模式 | 你的工作 | 耗时 | 适合 |
|---|---|---|---|
| **手动模式**(已废弃,见末尾 APPENDIX) | 9 阶段每段复制提示词、粘贴、检查 | 60-90 分钟 | 第一次想看清每一步发生什么 |
| **Swarm 模式(本指南默认)** | 输入 `/saas-case-swarm` + 在 4 个检查点拍板 | 20-30 分钟 | 真正办案、对结构已熟 |

**Swarm 模式下,你是主办律师,不是操作员。** Claude 主进程扮演 `legal-supervisor`(主办律师视角),自动拆任务、分派给 6 个 teammate(5 个生成/整理角色 + 1 个独立复核角色)、汇总成果、生成客户交付包。**你只在 4 个真正需要律师判断的关键节点上拍板**。

## 2. Swarm 架构(团队组成)

```text
                         ┌──────────────────────────┐
                         │   你(主办律师)         │
                         │   只在 4 个检查点拍板    │
                         └──────────┬───────────────┘
                                    │ AskUserQuestion
                                    ▼
                         ┌──────────────────────────┐
                         │ Supervisor               │  ← Claude 主进程
                         │ legal-supervisor.md      │     扮演这个角色
                         └──────────┬───────────────┘
                                    │ Agent tool spawn
              ┌──────────┬──────────┼──────────┬──────────┐
              ▼          ▼          ▼          ▼          ▼
        ┌─────────┐┌─────────┐┌─────────┐┌─────────┐┌─────────┐
        │intake-  ││contract-││evidence-││litigatio││client-  │
        │analyst  ││reviewer ││cataloger││nstrategi││packager │
        │         ││         ││         ││st       ││         │
        │Wave 1A  ││Wave 1B  ││Wave 2   ││Wave 3   ││Wave 5   │
        └─────────┘└─────────┘└─────────┘└─────────┘└─────────┘
                                                          ▲
                                          ┌─────────┐     │
                                          │risk-    │─────┘
                                          │reviewer │
                                          │Wave 4   │  独立复核全部上游产出
                                          │ 独立!   │
                                          └─────────┘
```

每个 teammate 都是 `Agent` 工具 spawn 的隔离子进程,**有自己的上下文窗口、各自只读取必要文件、产出后向 Supervisor 报告然后 idle**。其中部分角色读取 `.claude/agents/` 定义文件,`evidence-cataloger` 和 `client-packager` 由 Skill prompt 直接定义。这是真正的分工,不是同一个 Claude 反复扮演不同角色。

## 3. Wave 流水(发生什么)

| Wave | 谁干 | 输入 | 产出 | 并行? |
|---|---|---|---|---|
| **0 准备** | Supervisor | README / CLAUDE.md / 案件背景 | 团队 + 任务节点 | — |
| **1A 案件初始化** | intake-analyst | 01-客户原始材料/ 9 份 | `01-案件摘要.md` + `02-事实时间线.md` | ✅ 与 1B 并行 |
| **1B 合同审查** | contract-reviewer | 合同 + 补充协议 | `03-合同风险审查表.md` | ✅ 与 1A 并行 |
| **2 证据目录** | evidence-cataloger | Wave 1 全部 + 原始材料 | `04-证据目录.md` | 顺序 |
| **3 诉讼准备** | litigation-strategist | Wave 1+2 全部 | `05-诉讼准备报告.md` | 顺序 |
| **4 独立复核** | risk-reviewer | Wave 1-3 全部 5 份产出 | `06-复核报告.md` | 顺序 + **必须独立** |

## 4. 4 个人工检查点(你只在这里拍板)

### ① 启动前拍板(Wave 0 之后)

Supervisor 读完上下文,问你 3 件事:

| 问题 | 为什么必须问律师 |
|---|---|
| 立场默认代理 A 公司,确认还是切换? | AI 不能替你决定代理谁——这是利益冲突层的判断 |
| 优先目标:协商 / 诉讼 / 仲裁 / 看完报告再定? | 不同目标会让后续 brief 的语气和诉求结构不一样 |
| 任何特别指示? | 例如"避开特定主张""强调客户长期合作关系" |

### ② 策略拍板(Wave 3 之后)

Supervisor 把诉讼准备报告核心要点(3-5 行)给你,问:

| 问题 | 为什么必须问律师 |
|---|---|
| 4+ 个争议焦点中,认可哪几个? | AI 列的是可能性,**取舍是策略** |
| 诉求金额 200,000 元 / 违约金口径,同意还是调整? | 数字背后是商业判断,不是法律判断 |
| 先发律师函再起诉,还是直接立案? | 程序选择牵涉客户关系和成本,AI 不替你拍 |

### ③ 复核结果处置(Wave 4 之后)

Supervisor 显示 Risk Reviewer 找到的高/中风险问题清单,问:

| 选项 | 含义 |
|---|---|
| 全部修订 | Supervisor 调相关生成 agent 修一轮 → 再让 risk-reviewer 复核 → 最多 2 轮 |
| 只修高风险 | 中低风险接受 |
| 全部接受 | 跳过修订,直接进 Wave 5 |

**这一步本身就在演示 Reviewer 的核心价值**:它只挑问题、不替你决定怎么改。改不改、怎么改、还是直接接受——是主办律师的判断。

### ④ 客户版终审(Wave 5 之后)

Supervisor 把"给客户的话"邮件全文给你看,问:

| 问题 | 为什么必须问律师 |
|---|---|
| 客户版可以发吗 / 改哪段 / 退回重做? | 对外文件的最终责任在律师 |
| 邮件抄送给谁? | 客户沟通策略,AI 不替你决定 |

## 6. 监控与干预

### 实时查看进度

Supervisor 会在每个 Wave 完成时简短汇报(3-5 行)。你也可以随时:

```text
TaskList 一下
```

让 Supervisor 显示当前任务节点状态。

### 暂停或中止

- **暂停**:任何时候按 ESC 中断当前 turn,过程稿已经在 `04-工作过程/` 里,无损失
- **退出**:`/clear` 或 ctrl+C。重启后想继续就再次输入 `/saas-case-swarm`,告诉 Supervisor "从 Wave N 继续"

### 单独重跑某个 Wave

如果某个 Wave 产出不满意,在对话中直接对 Supervisor 说:

```text
Wave 3 的诉讼准备报告我不满意 —— 争议焦点太多、第 3 个删掉,然后重新 spawn litigation-strategist 跑一遍。
```

Supervisor 会 spawn 新的 teammate 重跑,不影响其他 Wave 的产出。

### 异常逃生口

任何 teammate 报告:
- "材料冲突"(例:合同金额与发票金额不一致)
- "信息不足"(例:违约金起算日依赖未提供的送达凭证)
- "超出能力"(例:涉及刑事问题,不在本工作流范围)

Supervisor 会**自动停下来**告诉你,不会继续推进。

## 7. 为什么这样设计(给好奇的你)

| 设计选择 | 理由 |
|---|---|
| Supervisor 不直接生成正文 | 角色单一性 — 主办律师统筹,不替专业 teammate 写稿 |
| Wave 1A/1B 强制并行 | 没有依赖关系的工作不该串行 — 这是 Swarm 区别于手动模式最大的加速 |
| Risk Reviewer 必须独立 | 生成与复核如果同源,会自我合理化 — 律师工作最大的隐性事故源 |
| 检查点只 4 个 | 多了变操作员,少了变签字机器 — 4 个是经验值,落在"真正非律师判断不可"的节点上 |
| Wave 6 复盘由 Supervisor 自己做 | 不需要 spawn 子进程 — 复盘本质是项目级总结,不是专业产出 |
| 修订循环上限 2 轮 | 防止 Reviewer-生成 来回死循环 — 超 2 轮意味着方向有问题,该律师介入 |
