# Lesson 20.2: 7 种产品演示类型——从概念演示到交互工作坊

## 本课目标

- 掌握 7 种产品演示类型的适用场景和结构
- 学会根据受众和目的选择最合适的演示类型
- 理解从故事到演示的完整落地流程

> **前置知识**：你已经在 Lesson 20 学会了故事讲述的 10 步工作流，在 Lesson 20.1 掌握了 26 个故事框架。本课是故事讲述的"最后一公里"——把故事变成用户、投资者、合作伙伴能亲眼看到的演示。

## 核心内容

### 演示类型总览

故事讲述的最终落地就是产品演示。WDS 区分了 7 种不同类型的演示，每种有不同目的：

| 类型 | 时长 | 核心目的 | 最佳观众 |
|------|------|---------|---------|
| 概念演示 | 5-10 分钟 | 展示核心概念 | 团队内部、早期投资者 |
| 功能演示 | 10-20 分钟 | 深入展示功能细节 | 潜在用户、销售团队 |
| 场景演示 | 15-30 分钟 | 真实情境中展示价值 | 用户研究小组、决策者 |
| 对比演示 | 10-15 分钟 | 展示相对优势 | 竞品用户、犹豫者 |
| 用户成功演示 | 15-25 分钟 | 展示真实成功案例 | 潜在用户、投资者 |
| 技术演示 | 20-30 分钟 | 展示技术优势 | CTO、系统集成商 |
| 交互工作坊 | 30-60 分钟 | 让用户亲手试用 | 培训对象、企业客户 |

---

### 1. 概念演示（Concept Demo）

**目的**：展示产品的核心概念

**形式**：简单动画 / 原型 / 交互视频，5-10 分钟

**观众**：团队内部、早期投资者、合作伙伴

**重点**：让人理解"这是什么"，不求面面俱到

**例**：
```
"这是我们的 AI 记账工具。
它解决的核心问题是：
小微企业主忘记记账，月底对不上账。

核心创新是：
用 AI 自动识别发票，减少 90% 的手动输入。

演示：给一张发票照片 → 1秒识别→金额+分类自动填好"
```

---

### 2. 功能演示（Feature Demo）

**目的**：深入展示某个功能的细节

**形式**：完整的操作流程录屏或现场演示，10-20 分钟

**观众**：潜在用户、合作伙伴、销售团队

**重点**：展示"怎么用"，涵盖主要交互流程

**例**：
```
演示 AI 分类功能：
1. 打开 App → 点击"记账"
2. 拍发票照片
3. 系统识别：
   - 金额：¥156
   - 商家：XX 便利店
   - 分类：原材料采购（自动）
4. 用户可编辑分类
5. 点击"保存"
6. 进入记录详情页
```

---

### 3. 场景演示（Scenario Demo）

**目的**：在真实情境中展示产品如何帮助用户

**形式**：角色扮演 / 视频故事 / 交互原型走查，15-30 分钟

**观众**：潜在用户、用户研究小组、客户决策者

**重点**：展示"为什么有用"，从用户的心理痛点出发

**例**：
```
"我是一个早餐店老板，每天 50-100 笔交易。

现在的问题：
- 手写账本容易漏写
- 月底一次记 500+ 笔非常烦
- 经常对不上账

演示场景：
[晚上 11 点，打烊后]
1. 掏出手机，打开我们的 App（而不是翻账本）
2. 快速拍今天的所有收据
3. App 批量识别和分类
4. 3 分钟完成，对比原来的 45 分钟
5. 看到今天营收 ¥4,200，自动分类统计
6. 心里踏实了，可以回家睡觉了
"
```

---

### 4. 对比演示（Comparison Demo）

**目的**：展示相对于竞品或传统方式的优势

**形式**：并排对比视频 / AB 演示，10-15 分钟

**观众**：竞争对手的用户、犹豫者

**重点**：让用户看到具体的改进（时间、步骤、准确性）

**例**：
```
对比：手写账本 vs 我们的 AI 工具

[分屏演示]

左边：传统方式
- 打开账本（找了 1 分钟）
- 翻到今天的页面（又找了 30 秒）
- 手写：日期、金额、说明、分类（1 分钟）
- 总耗时：2.5 分钟 / 笔

右边：AI 工具
- 打开 App（5 秒）
- 拍发票（3 秒）
- AI 自动填好（2 秒）
- 总耗时：10 秒 / 笔

结论：快 15 倍，准确率 99% vs 60%
```

---

### 5. 用户成功故事演示（User Success Demo）

**目的**：展示真实用户的成功案例

**形式**：用户访谈 + 产品使用截屏，15-25 分钟

**观众**：潜在用户、投资者、合作伙伴

**重点**：增加可信度，让新用户看到"和我一样的人已经成功了"

**例**：
```
用户：李女士，开了一个淘宝店，月销 50 万

问题：原来用 Excel 管理，越来越复杂
      "有时候一个数字算错，要查一小时"

使用产品后（3 个月）：
- 实现了财务自动化
- 识别出最赚钱的商品品类
- 根据数据调整库存策略
- 月利润提高 15%

她的话："这个工具让我能把更多时间花在运营上，
         而不是陷入数据的细节。"

[展示她在 App 中的实时数据和分析报告]
```

---

### 6. 技术/架构演示（Technical Demo）

**目的**：展示技术方面的优势（特别是 B2B 或合作场景）

**形式**：技术流程图 / 代码展示 / 性能对比，20-30 分钟

**观众**：技术决策者、CTO、系统集成商

**重点**：强调可靠性、扩展性、集成能力

**例**：
```
"我们的 AI 识别引擎基于：
- 100 万+ 发票训练数据
- 99.2% 的识别准确率
- 100ms 的单张识别延迟

集成方式：
- 1. REST API（最灵活）
- 2. SDK（最快速）
- 3. Webhook（最异步）

SLA：
- 99.99% 可用性
- 支持日均 100 万+ API 调用"
```

---

### 7. 交互式工作坊演示（Workshop Demo）

**目的**：让用户亲手试用产品，互动式学习

**形式**：小组动手操作 + 讲师引导，30-60 分钟

**观众**：用户体验研究小组、企业客户、培训对象

**重点**：体验，而非展示

**例**：
```
"现在我们进入动手环节。

每个人拿出一张真实的发票。
打开我们的 App。
跟我一起：
1. 点击'+'按钮
2. 拍照你的发票
3. 看 AI 怎么自动填好

...
然后：
- 你可以编辑分类
- 看看这个月的财务概览
- 下载一个自动报表

有什么疑问吗？"
```

### 选择合适的演示类型

```
使用场景            → 推荐演示类型
────────────────────────────────────
初创融资演讲         → 概念演示 + 用户故事演示
销售与潜在客户      → 场景演示 + 对比演示
客户成功案例        → 用户成功故事演示
竞争分析            → 对比演示
员工培训            → 交互式工作坊演示
技术合作谈判        → 技术架构演示
产品发布会          → 概念演示 + 多个场景演示 + 用户故事
内部团队对齐        → 概念演示 + 功能演示
```

### 从故事到演示的完整流程

```
Step 1: 定义演示目的
        ↓ 我要向谁演示？想让他们做什么？

Step 2: 选择演示类型
        ↓ 根据目的选择 7 种中的 1-2 种

Step 3: 从 Trigger Map 和 UX 场景提取故事
        ↓ 用这些真实的设计数据作为故事基础

Step 4: 用 26 个框架选择故事类型（Lesson 20.1）
        ↓ 例：用"对比故事"讲解"用前 vs 用后"

Step 5: 执行 10 步故事讲述工作流（Lesson 20）
        ↓ 从规划→构思→细节→交付

Step 6: 准备演示资料
        ↓ 原型、视频、截屏、数据等

Step 7: 试讲和反馈
        ↓ 在小组中反复练习，根据反馈调整

Step 8: 正式演讲
        ↓ 记住：故事比功能列表更有力
```

## 常见问题

**Q: 演示要多长时间最好？**

A: 取决于观众和目的。融资演讲 5-10 分钟足够（留时间问答）。销售演示 15-20 分钟。内部分享 30 分钟。原则是：**不要让观众走神**。如果观众开始看手机，说明你讲得太长或太无聊。

**Q: 产品还没开发完成，能演示吗？**

A: 可以。用原型、线框图、视频模拟都可以。关键是"故事要真"——即使产品还是概念，用户的痛点和需求是真实的。好的演示是"这个概念能解决的真实问题"，而不是"这个产品已经完美"。

**Q: 一次演示可以混合多种类型吗？**

A: 可以。比如产品发布会：开场用"概念演示"（3 分钟），中间用"场景演示"（10 分钟），结尾用"用户成功故事"（5 分钟）。混合使用能让演示节奏更丰富。
