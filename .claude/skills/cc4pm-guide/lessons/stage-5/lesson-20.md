# Lesson 20: 团队落地指南

## 本课目标

- 掌握 cc4pm 的安装和配置方法
- 了解从个人使用到团队推广的渐进策略
- 理解不同角色（PM、工程师、设计师、QA）各自使用什么
- 建立一个 5 周渐进式采用计划

## 核心内容

### 从"我学会了"到"团队用起来"

恭喜你走到了最后一课。前面 19 课你学了很多——CIS 创意方法、BMM 产品工作流、WDS 设计方法论、工程质量工具、知识沉淀系统。

但学会和用起来之间还有一道鸿沟：**团队采纳**。

好工具推不下去的三大原因：

```
1. "太复杂了，我不想学"     → 需要渐进式引入
2. "我现在的方式也能做"     → 需要对比价值展示
3. "老板/团队不支持"        → 需要从小处证明价值
```

这课教你怎么绕过这三个坑。

### 安装和配置

#### 基础安装

cc4pm 的安装非常简单——它是一个 Claude Code 插件，一条命令搞定：

```bash
# 使用 npm 安装（推荐）
npm install -g cc4pm

# 或者克隆仓库
git clone https://github.com/anthropics/cc4pm.git ~/.claude
```

安装完成后，所有的代理、技能、命令就自动可用了。

#### 按角色配置

不同角色需要的模块不同，可以选择性安装：

| 角色 | 必装模块 | 可选模块 |
|------|---------|---------|
| PM | CIS + BMM | WDS |
| 工程师 | 工程工具链 | BMM（理解需求） |
| 设计师 | WDS | CIS（故事讲述） |
| QA | 工程工具链 | BMM（理解验收标准） |
| 技术负责人 | 全部 | - |

#### 环境配置

```bash
# 设置包管理器（如果不是 npm）
export CLAUDE_PACKAGE_MANAGER=pnpm  # 或 yarn, bun

# 设置语言偏好
# 在项目根目录创建 _bmad/cis/config.yaml
communication_language: zh-CN
```

### 推广策略：三步走

#### 第一步：个人使用（1-2 周）

> "先自己用起来，成为团队的'首席体验官'"

```
目标：你自己熟练使用 cc4pm 的核心功能
方法：在日常工作中用 cc4pm 替代传统方式
证据：收集"效率提升"的对比数据

示例：
- 用 /bmad-brainstorming 做头脑风暴
  → 对比：传统方式30分钟20个想法 vs cc4pm方式30分钟100+想法
- 用 /bmad-create-prd 写 PRD
  → 对比：传统方式3天 vs cc4pm方式3小时
- 用 /plan 做实现规划
  → 对比：传统方式靠经验估算 vs cc4pm自动分析代码库
```

#### 第二步：小团队试点（2-4 周）

> "找 2-3 个同事一起用，形成小团体"

```
目标：验证 cc4pm 在团队协作中的价值
方法：
- 找一个真实项目做试点
- 用 cc4pm 走一遍完整流程
- 记录效率数据和使用体验

试点团队的理想组成：
- 1 个 PM（你）
- 1 个工程师
- 1 个设计师（可选）

试点项目的选择标准：
- 不是特别紧急（留有学习空间）
- 不是特别简单（能体现工具价值）
- 有明确的需求和验收标准
```

#### 第三步：全团队推广（持续）

> "用试点结果说服更多人"

```
目标：全团队日常使用 cc4pm
方法：
- 展示试点项目的效率数据
- 组织分享会，试点成员现场演示
- 从最有价值的命令开始推（/plan, /code-review）
- 不要一次推所有功能——循序渐进
```

### 5 周渐进式采用路径

如果你要在团队中推 cc4pm，这是推荐的时间表：

#### 第一周：安装 + /plan

```
目标：让团队体验"AI 辅助规划"的价值
做什么：
- 全团队安装 cc4pm
- 拿一个真实需求运行 /plan
- 对比人工估算和 /plan 估算的差异
预期收获："原来 AI 真的能帮我做规划"
```

#### 第二周：BMM 头脑风暴 + PRD

```
目标：让 PM 体验"AI 辅助产品设计"
做什么：
- 用 Carson（头脑风暴）为一个新需求做创意发散
- 用 John（PRD 编辑）创建一份完整 PRD
- 对比传统方式的时间和质量
预期收获："3小时完成了以前需要3天的工作"
```

#### 第三周：CIS 创新策略

```
目标：让团队体验"系统化创新"
做什么：
- 用 Victor（创新策略师）做一次竞争分析
- 尝试蓝海策略分析或 Jobs-to-be-Done 分析
- 把结果带到团队会议上讨论
预期收获："这种分析维度是我自己想不到的"
```

#### 第四周：WDS Trigger Map

```
目标：引入"用户心理驱动设计"
做什么：
- 用 Saga 为一个功能做 Trigger Map
- 对比"功能驱动"和"心理驱动"的需求优先级差异
- 让设计师参与 Freya 的 UX 场景设计
预期收获："原来功能优先级可以不用吵架"
```

#### 第五周及持续：/learn + /evolve 知识沉淀

```
目标：建立团队的"AI 增强记忆"
做什么：
- 每个人在解决问题后运行 /learn
- 每周做一次 /evolve 分析
- 有价值的经验用 /promote 推广到全团队
- 定期清理过时的本能
预期收获："AI 越来越懂我们的工作方式了"
```

### 成功指标和评估方法

推广 cc4pm 后，怎么衡量效果？

| 指标 | 衡量方式 | 目标 |
|------|---------|------|
| 规划效率 | 从需求到计划的时间 | 缩短 50%+ |
| PRD 质量 | 需求评审的修改次数 | 减少 60%+ |
| 开发效率 | Story 完成时间 | 缩短 30%+ |
| 代码质量 | 上线后 Bug 数 | 减少 50%+ |
| 团队满意度 | 匿名问卷 | > 7/10 |
| 知识复用率 | /learn 提取的经验被引用次数 | 每周 3+ 次 |

### 常见阻力和应对策略

#### 阻力 1："我不信 AI 写的东西"

```
应对：
- 不要说"AI写的比你好"——这会引起防御心理
- 说"AI帮你做初稿，你做最终审核"
- 强调 AI 是"增强"不是"替代"
- 让他们先看 /plan 的输出——通常质量会让他们惊讶
```

#### 阻力 2："学这个太花时间了"

```
应对：
- 不要一次教所有功能——每周只教一个命令
- 从最痛的点切入（比如 PM 最头疼的 PRD 写作）
- 做 15 分钟的午间分享，不要做 2 小时的培训
- 准备"抄袭模板"——copy-paste 就能用的命令示例
```

#### 阻力 3："我的方式也能做"

```
应对：
- 不否定他们的方式——承认"你的方式也可以"
- 但展示效率差异：数据说话
  "你的 PRD 写了3天，我的用 cc4pm 3小时搞定，质量差不多"
- 找到他们最痛的点，展示 cc4pm 的解法
- 邀请他们旁观你用 cc4pm 工作
```

#### 阻力 4："老板不支持"

```
应对：
- 不要先问老板——先自己用出成果
- 用成果说话：
  "上个 Sprint 我用 cc4pm 做规划，
   估算偏差从30%降到10%，
   PRD返工从3次降到0次"
- 找机会让老板看到 cc4pm 的输出（比如一份漂亮的竞争分析报告）
- 如果可能，用 /bmad-cis-storytelling 为 cc4pm 推广写一个故事
```

## 常见问题

**Q: 如果我只能记住 3 个命令，应该记哪 3 个？**

A: `/plan`（理解实现复杂度和排期）、`/bmad-brainstorming`（激发创意）、`/save-session`（保存工作上下文）。这三个覆盖了 PM 最高频的场景。

**Q: 团队有人用其他 AI 工具（比如 Cursor），冲突吗？**

A: 不冲突。cc4pm 是 Claude Code 的插件，和 Cursor 等工具在不同层面工作。cc4pm 提供的是"方法论和工作流"——代理、技能、命令；Cursor 提供的是"代码编辑体验"。它们可以共存。

**Q: cc4pm 有多少个代理/命令/技能？**

A: 截至目前：33 个 AI 代理、48+ 命令、96+ 技能。但你不需要全部掌握——80% 的价值来自 20% 的核心功能。本课程教的就是那 20%。

**Q: 学完这 20 课，还需要继续学什么？**

A: 实战是最好的学习。建议：(1) 在真实项目中使用，遇到问题再查具体命令的帮助文档；(2) 关注 cc4pm 的更新日志，了解新功能；(3) 参与社区讨论，学习其他用户的最佳实践。

**Q: 20 课的内容太多了，有没有速查表？**

A: 你现在读的就是结构化的知识——每课的"本课目标"和"常见问题"就是最好的速查表。建议收藏这 20 课的目录，需要时回来查阅具体章节。
