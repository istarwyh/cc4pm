# Lesson 11: WDS 概览：从用户心理到设计规范

## 本课目标

- 理解 WDS（Web Design System）的核心理念：以用户心理为驱动的设计方法
- 掌握 WDS 的 8 阶段设计流程全景
- 认识两个关键设计代理：Saga（故事女神）和 Freya（UX 设计师）
- 理解 WDS 与传统 UX 设计的本质区别

## 核心内容

### 什么是 WDS？

先问你一个问题——

> 你的产品经理做需求时，是先想"用户会点哪个按钮"，还是先想"用户为什么要打开这个 App"？

如果是前者，你正在做"界面设计"。如果是后者，恭喜，你已经有了 WDS 的思维。

WDS（Web Design System）不是一套 UI 组件库，也不是一个设计工具——它是一套**以用户心理为驱动、从战略到像素的 8 阶段设计方法论**。

一句话定义：

> **WDS 是一套从"用户为什么来"到"每个像素如何呈现"的完整设计系统。**

### 传统 UX vs WDS：核心区别

| 维度 | 传统 UX 设计 | WDS 方法 |
|------|-------------|---------|
| 起点 | "用户要什么功能" | "用户的心理驱动力是什么" |
| 流程 | 需求 -> 线框图 -> UI | 心理触发 -> 场景 -> 概念规范 -> 设计系统 |
| 决策依据 | "竞品这么做" / "我觉得" | 每个决策都有记录的心理学理由 |
| 文档 | Figma 文件 + 零散备注 | Design Log 完整追溯每个决策 |
| 协作方式 | 设计师画完扔给开发 | 两个 AI 代理结构化协作 |

说白了，传统 UX 设计经常是"拍脑袋"——而 WDS 的每一个设计决策都有**可追溯的理由**。

### WDS 的 8 阶段流程

WDS 把整个设计过程拆成 8 个阶段，每个阶段都有明确的输入、输出和负责的 AI 代理：

```
阶段 0: 项目设置（Project Setup）
↓ 创建项目结构和配置文件

阶段 1: 产品简报（Product Brief）       ← Saga 负责
↓ 定义愿景、定位、目标用户

阶段 2: Trigger Map（心理触发映射）      ← Saga 负责
↓ 从心理驱动力映射到功能需求

阶段 3: UX 场景（Scenarios）            ← Freya 负责
↓ 将 Trigger Map 转化为用户旅程

阶段 4: 概念规范（Conceptual Specs）     ← Freya 负责
↓ 每个页面、每个状态的完整描述

阶段 5: 设计系统（Design System）        ← Freya 负责
↓ 颜色、字体、组件的完整规范

阶段 6: 线框图（Wireframes）            ← Freya 负责
↓ 页面布局和交互设计

阶段 7: 设计交付（Design Delivery）      ← Freya 负责
↓ 确保规范完整性，移交开发
```

**关键洞察**：注意到了吗？在你画第一个线框图之前，你已经走了 4 个阶段——这 4 个阶段全都在回答"为什么"和"做什么"，而不是"怎么做"。

### 认识你的两位设计伙伴

#### Saga（故事女神）—— 战略分析师

Saga 是 WDS 的第一位代理，她的定位是**战略分析师 + 研讨会引导者**。

**Saga 的核心职责**：

- 引导产品简报的创建（愿景、定位、商业模式）
- 主持 Trigger Map 研讨会（用户心理 -> 功能映射）
- 从已有文档中提取战略信息
- 做"灵感分析"——研究竞品和参考设计

**Saga 的三种工作模式**：

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| Workshop（研讨会） | 一步步引导你完成分析 | 第一次做，想深度参与 |
| Suggest（建议） | Saga 先给建议，你审核调整 | 有基础材料，想加速 |
| Dream（AI自主） | Saga 自主完成，你最后审核 | 时间紧张，信任 AI |

**启动 Saga**：

```bash
/bmad-agent-wds-saga-analyst
```

#### Freya（UX 设计师）—— 设计执行者

Freya 是 WDS 的第二位代理，她的定位是**UX 设计师 + 规范制定者**。

**Freya 的核心职责**：

- 将 Trigger Map 转化为 UX 场景和用户旅程
- 创建概念规范（每个元素、每个状态的完整描述）
- 设计设计系统（颜色、字体、间距、组件）
- 生成线框图和页面设计
- 确保设计交付的完整性

**Freya 的工作特点**：

- 同样支持 Discuss、Suggest、Dream 三种模式
- 她的每个设计决策都会记录在 Design Log 中
- 她会自动引用 Trigger Map 中的心理触发因素来解释设计选择

**启动 Freya**：

```bash
/bmad-agent-wds-freya-ux
```

### Design Log：让每个决策都有据可查

WDS 最让 PM 兴奋的一个设计就是 **Design Log（设计日志）**。

传统设计中，你问设计师"这个按钮为什么放在这里？"，得到的答案通常是"因为好看"或"因为用户习惯"。在 WDS 中，每一个设计决策都会被记录：

```markdown
## Design Log Entry

### 决策：首页使用全屏 Hero 布局
- 触发因素：用户角色"初次访问者"的心理需求是"快速理解产品价值"
- Trigger Map 引用：TM-3.2 "信任建立"触发因素
- 替代方案：列表式布局（被否决，原因：不够有冲击力）
- 影响范围：首页、着陆页
- 记录人：Freya
- 日期：2026-03-22
```

**PM 为什么要关心 Design Log**：

1. **需求追溯**：每个设计都能追溯到具体的用户需求和心理驱动
2. **沟通利器**：给老板/利益相关方解释设计决策时有理有据
3. **迭代依据**：改设计不是"我觉得不好看"，而是"这个触发因素的优先级变了"

### 微文件架构：WDS 的技术亮点

WDS 采用了一种叫"微文件架构"的组织方式——每个工作流步骤都是一个独立的、自包含的 Markdown 文件。

**举个例子**——Trigger Map 工作流的文件结构：

```
2-trigger-mapping/
├── workflow.md                           # 工作流主入口
├── workflow-validate.md                  # 验证工作流
├── steps-c/
│   ├── step-00a-documentation-synthesis.md   # 从已有文档提取
│   ├── step-01-overview.md                   # 概述和模式选择
│   ├── step-02-business-goals.md             # 业务目标定义
│   ├── step-03-user-roles.md                 # 用户角色定义
│   ├── step-04-driving-forces.md             # 心理驱动力
│   └── step-05-scoring.md                    # 评分矩阵
└── data/
    ├── business-goals-template.md            # 模板
    └── quality-checklist.md                  # 质量检查清单
```

**微文件架构的好处**：

- **上下文管理**：AI 每次只加载当前步骤的文件，不会被海量信息淹没
- **可追溯**：每个步骤的输入输出都是明确的
- **可复用**：每个步骤文件可以独立使用和修改
- **质量保证**：每个步骤都有内置的质量检查

### WDS 对 PM 的价值

你可能会问——"我又不画界面，学 WDS 有什么用？"

看看这张表：

| PM 的日常痛点 | WDS 怎么解决 |
|-------------|------------|
| "设计师做的不是我想要的" | Trigger Map 把需求精确到心理驱动力层面 |
| "老板问我这个功能为什么这么设计" | Design Log 提供完整的决策追溯 |
| "用户不用我们的产品" | 从用户心理出发，而不是从功能列表出发 |
| "需求优先级怎么排" | Trigger Map 的评分矩阵自动排序 |
| "设计改了又改没完没了" | 每个设计都锚定在心理触发因素上，改动有理有据 |

## 常见问题

**Q: WDS 只适合 Web 产品吗？**

A: 名字叫 Web Design System，但核心方法论——心理驱动设计——适用于任何数字产品。Trigger Map 可以用于 App、小程序、SaaS、甚至硬件产品的界面设计。核心是"从用户心理出发"的思维方式，不是具体的技术实现。

**Q: WDS 能替代 Figma 吗？**

A: 不能，也不打算替代。WDS 产出的是设计规范和概念文档——它告诉你"做什么、为什么做"。Figma 负责"怎么做"——把规范变成视觉设计。两者是互补关系，WDS 的输出是 Figma 的输入。

**Q: Saga 和 Freya 的分工会不会有冲突？**

A: 不会，因为她们的职责边界很清晰。Saga 负责"战略层"（阶段 1-2：产品简报和 Trigger Map），Freya 负责"执行层"（阶段 3-7：场景、规范、设计系统、线框图、交付）。她们之间通过 Trigger Map 文档做交接——Saga 的输出就是 Freya 的输入。

**Q: 我已经有产品了，还需要从阶段 0 开始吗？**

A: 不需要。WDS 支持"从已有文档提取"模式——Saga 可以读取你现有的 PRD、竞品分析、用户研究报告，从中提取信息来构建 Trigger Map。你不需要推倒重来。
