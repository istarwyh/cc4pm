# Lesson 18: Trigger Map：用户心理→功能映射

## 本课目标

- 掌握 Trigger Map（触发力地图）的核心思想：从"用户为什么"驱动功能设计
- 理解 4 个结构化研讨会（Workshop）的完整流程和产出
- 掌握正向驱动力（Positive Drivers）和负向驱动力（Negative Drivers）的区别和应用
- 学会使用功能影响分析（Feature Impact Analysis）对功能优先级排序

## 核心内容

### 什么是 Trigger Map？

**一句话解释**：Trigger Map（触发力地图）是一种把"商业目标"映射到"用户心理驱动力"再映射到"具体功能"的结构化方法。

**用一个比喻来理解**：

```
传统功能设计：
  "我觉得用户需要这个功能" → 开发 → 发现没人用

Trigger Map 的做法：
  "用户为什么会打开我们的 App？"
  → "因为他害怕忘记记账导致月底对不上账"（负向驱动力）
  → "因为他想看到自己的攒钱进度"（正向驱动力）
  → 基于这些心理，设计对应的功能
```

### Trigger Map 的四层结构

Trigger Map 是 Effect Mapping（效果映射）的改良版。它有 4 层结构，从上到下逐层展开：

```
Layer 1: 商业目标（Business Goals）
         ↓ "我们想达成什么？"

Layer 2: 产品（Product）
         ↓ "用什么产品达成？"

Layer 3: 目标用户群（Target Groups）
         ↓ "谁在使用这个产品？"

Layer 4: 驱动力（Driving Forces）
         ↓ "用户为什么要用？"
         ├── 正向驱动力（Positive Drivers）：用户"想要"的
         └── 负向驱动力（Negative Drivers）：用户"害怕"的
```

**为什么要区分正向和负向驱动力？**

这来自于行为心理学中的"趋避动机"理论：

```
正向驱动力（Toward Motivation）：
  "我想要..."
  例：我想看到自己的财务进步 → 设计：进度可视化

负向驱动力（Away-from Motivation）：
  "我害怕/不想..."
  例：我害怕漏记一笔账 → 设计：智能提醒

核心原理——损失厌恶（Loss Aversion）：
  人们对"失去"的恐惧是"获得"快乐的 2 倍。
  负向驱动力往往比正向驱动力更有力。
```

### 驱动力的 WHAT + WHY + WHEN 模式

每个驱动力都用三个维度来描述，确保足够具体：

```
WHAT（什么）：驱动力的具体内容
WHY （为什么）：这个驱动力为什么重要
WHEN（什么时候）：在什么情境下触发

示例——正向驱动力：
  WHAT：看到每月支出趋势图
  WHY ：感觉自己在掌控财务
  WHEN：月末查看账单时

示例——负向驱动力：
  WHAT：担心忘记记录现金支出
  WHY ：月底对账发现差额会焦虑
  WHEN：外出消费用现金付款后
```

### 4 个研讨会（Workshop）详解

Trigger Map 的创建过程由 4 个结构化研讨会组成。每个研讨会有明确的目标和产出。

#### Workshop 1: 目标与产品对齐

**目标**：确定商业目标和产品定位，建立 Trigger Map 的第 1-2 层。

**流程**：

```
Step 1: 明确商业目标
        ↓ 你的产品要达成什么商业结果？
        ↓ 例：6个月内获得 10,000 付费用户

Step 2: 定义产品范围
        ↓ 用什么产品形态来达成这个目标？
        ↓ 例：移动端 AI 记账 App

Step 3: 确认衡量标准
        ↓ 怎么知道目标达成了？
        ↓ 例：月活跃用户数、付费转化率
```

**产出**：商业目标清单 + 产品定位文档

**启动方式**：

```bash
/bmad-wds-trigger-mapping
# Saga 会引导你进入 Workshop 1
```

#### Workshop 2: 目标用户群分析

**目标**：识别和描述所有目标用户群，建立 Trigger Map 的第 3 层。

**流程**：

```
Step 1: 列出所有可能的用户群
        ↓ 谁会使用这个产品？
        ↓ 例：小微企业主、自由职业者、家庭记账用户

Step 2: 为每个用户群创建画像
        ↓ 他们的特征、需求、痛点是什么？
        ↓ 例：小微企业主——35岁，开餐厅，每天100+笔交易

Step 3: 确定优先级
        ↓ 先服务谁？
        ↓ 例：MVP 先服务小微企业主

Step 4: 描述使用情境
        ↓ 用户在什么场景下使用产品？
        ↓ 例：晚上打烊后整理当天收支
```

**产出**：用户群画像 + 优先级排序 + 使用情境

#### Workshop 3: 驱动力发现

**这是 4 个研讨会中最关键的一个**——发现用户的正向和负向心理驱动力。

**目标**：为每个用户群，挖掘他们使用产品的深层心理动机。

**流程**：

```
Step 1: 正向驱动力头脑风暴
        ↓ "用户想要从产品中得到什么？"
        ↓ 使用 WHAT + WHY + WHEN 格式

Step 2: 负向驱动力头脑风暴
        ↓ "用户害怕什么？不想发生什么？"
        ↓ 使用 WHAT + WHY + WHEN 格式

Step 3: 驱动力分类和合并
        ↓ 去重、分组、标注关联关系

Step 4: 驱动力强度评估
        ↓ 哪些驱动力更强烈？
        ↓ 提示：负向驱动力通常更强
```

**示例——AI 记账工具的驱动力**：

```
小微企业主的正向驱动力：
  [P1] 看到清晰的利润趋势 → 感觉生意在变好 → 月末复盘时
  [P2] 一键生成税务报表   → 省去找会计的时间 → 季度报税前
  [P3] 了解同行业平均水平 → 知道自己做得怎么样 → 制定下季度目标时

小微企业主的负向驱动力：
  [N1] 担心漏记收入导致税务问题 → 罚款和法律风险 → 每天结束营业时
  [N2] 害怕现金流断裂没有预警   → 突然发不出工资 → 每月中旬
  [N3] 不想花太多时间在记账上   → 时间应该用在经营上 → 每天
```

#### Workshop 4: 功能影响分析（Feature Impact Analysis, FIA）

**目标**：把驱动力转化为具体功能，并评估每个功能对驱动力的影响程度。

**流程**：

```
Step 1: 为每个驱动力设计对应功能
        ↓ "什么功能能满足/缓解这个驱动力？"

Step 2: 建立功能-驱动力映射矩阵
        ↓ 每个功能影响哪些驱动力？强度如何？

Step 3: 功能评分
        ↓ 根据影响的驱动力数量和强度排序

Step 4: 确定 MVP 功能集
        ↓ 选出影响最大的功能作为第一版
```

**功能-驱动力映射矩阵示例**：

```
                    [P1] [P2] [P3] [N1] [N2] [N3]  总分
智能记账提醒          -    -    -    ★★★   ★    ★★   = 9
一键报表生成          ★    ★★★  -    ★★    -    ★★   = 11
AI 自动分类           -    ★    -    ★★    -    ★★★  = 9
现金流预测            ★★   -    ★    -    ★★★   -    = 9
行业对标分析          -    -    ★★★  -     -    -    = 4

★ = 1分，★★ = 2分，★★★ = 3分

MVP 优先级：一键报表生成 > 智能记账提醒 = AI自动分类 = 现金流预测 > 行业对标
```

**为什么这个方法比"拍脑袋定优先级"好**：因为每个功能的优先级都有心理学依据——它影响了多少个用户驱动力、影响强度多大。这不是"我觉得这个重要"，而是"数据显示用户的心理需求指向这个功能"。

### Trigger Map 的完整产出

4 个研讨会完成后，你会得到一套完整的文档：

```
trigger-map/
├── business-goals.md           # 商业目标
├── target-groups.md            # 目标用户群画像
├── driving-forces.md           # 正向/负向驱动力清单
├── feature-impact-analysis.md  # 功能影响分析矩阵
└── trigger-map-summary.md      # Trigger Map 汇总文档
```

### Trigger Map 的质量验证

创建完 Trigger Map 后，可以用验证工作流检查质量：

```bash
/bmad-wds-trigger-mapping-validate
```

**验证标准**：

| 检查项 | 标准 | 常见问题 |
|--------|------|---------|
| 商业目标 | 具体、可衡量 | "做大做强"不是目标 |
| 用户群 | 有具体画像 | "所有人"不是用户群 |
| 驱动力 | 有 WHAT+WHY+WHEN | 只写了 WHAT 没写 WHY |
| 正负平衡 | 同时有正向和负向 | 只想了正向忘了负向 |
| 功能映射 | 每个功能至少对应一个驱动力 | 有功能找不到驱动力来源 |
| 覆盖率 | 每个驱动力至少有一个功能 | 有驱动力没有对应功能 |

### 实操：为"AI 记账工具"创建 Trigger Map

**前提**：你已经用 BMM 创建了 PRD，现在要进入 WDS 的 Trigger Map 阶段。

**步骤 1：启动 Saga**

```bash
/bmad-wds-trigger-mapping
```

**步骤 2：Workshop 1 — 确认目标**

```
Saga：让我们开始 Trigger Mapping。
首先，你的产品要达成什么商业目标？

你：我们的 AI 记账工具，6个月内要获得 10,000 付费用户，
月均使用频次不低于 20 次。

Saga：好的。目标很清晰。我们的产品形态是？

你：移动端 App + Web 端管理后台。

Saga：明白了。Workshop 1 完成。
商业目标：10K 付费用户/6月 + 月活 20次
产品形态：Mobile App + Web Dashboard
```

**步骤 3：Workshop 2 — 用户群分析**

```
Saga：现在来分析目标用户群。
谁会使用这个记账工具？

你：主要是小微企业主，特别是开店的。
其次是自由职业者。

Saga：开店的小微企业主——具体说说。
他们的日常是什么样的？记账时最痛苦的是什么？

你：每天100+笔交易，大部分是小额的。
最痛苦的是月底对账总是对不上。

Saga：很好。让我继续追问——
他们现在用什么记账？Excel？手写？其他 App？
...
```

**步骤 4：Workshop 3 — 发现驱动力**

```
Saga：现在是最关键的环节——驱动力发现。

先想正向的：小微企业主使用记账工具时，
他最"想要"什么？什么会让他觉得"这个 App 真好用"？

你：他想看到这个月比上个月赚得多。

Saga：很好。用 WHAT+WHY+WHEN 描述——
WHAT：看到月度利润增长趋势
WHY ：获得"生意在变好"的确认感
WHEN：月末复盘时

现在想负向的：他最"害怕"什么？
...
```

**步骤 5：Workshop 4 — 功能优先级**

```
Saga：基于这些驱动力，我生成了功能影响分析矩阵。

最高分功能：
1. 一键报表生成（11分）—— 同时满足正向和负向驱动力
2. 智能记账提醒（9分）—— 主要缓解负向驱动力
3. AI 自动分类（9分）—— 节省时间的核心功能

建议 MVP 功能集：以上三个 + 基础记账功能

你觉得这个优先级合理吗？
```

### Trigger Map 与后续阶段的衔接

Trigger Map 不是独立存在的——它是 WDS 后续所有阶段的基础：

```
Trigger Map 的驱动力
    ↓
Phase 3: UX 场景 → 每个场景基于一个或多个驱动力
    ↓
Phase 4: UX 设计 → 每个界面元素服务于某个驱动力
    ↓
Phase 5: 开发    → 每个功能可以追溯到驱动力来源
    ↓
Phase 8: 演化    → 根据驱动力满足度决定改进方向
```

**这就是 WDS 最强大的地方**：从 Trigger Map 开始，每一个设计和开发决策都有心理学依据，而不是"我觉得这样好"。

## 常见问题

**Q: Trigger Map 和用户画像（Persona）有什么区别？**

A: 用户画像描述的是"用户是什么样的人"（年龄、职业、习惯），Trigger Map 描述的是"用户为什么要用你的产品"（心理驱动力）。用户画像是静态的人物描写，Trigger Map 是动态的心理分析。两者互补——先做用户画像定义"是谁"，再做 Trigger Map 分析"为什么"。

**Q: 4 个研讨会需要多长时间？**

A: 一个中等复杂度的产品，4 个研讨会大约需要 1-2 小时。Workshop 3（驱动力发现）通常花时间最多，因为挖掘深层心理需要反复追问。Saga 会帮你引导这个过程——你只需要回答问题和确认发现。

**Q: 驱动力想不出来怎么办？**

A: Saga 有一套追问技巧帮你挖掘。常用方法：(1) 回忆你自己作为用户的体验——"上次你记账时最烦的是什么？"；(2) 用"5个为什么"追问——"为什么对账对不上很烦？""因为要花时间找差额""为什么这让你烦？""因为这个时间可以用来经营"；(3) 参考竞品评论——用户在差评里说的就是负向驱动力。

**Q: Trigger Map 可以在开发过程中修改吗？**

A: 可以，而且应该修改。随着你对用户理解的加深，驱动力会更新、功能优先级会变化。关键是每次修改后，检查下游的 UX 场景和设计是否也需要同步更新——这就是 BMM 路线纠正（CC, Correct Course）工作流的价值所在。
