# Lesson 13: UX 场景与用户旅程设计

## 本课目标

- 理解从 Trigger Map 到 UX 场景的转化逻辑
- 掌握 Freya UX 设计师的三种工作模式
- 理解概念规范（Conceptual Specs）的作用和格式
- 实操设计一个完整的用户注册流程

## 核心内容

### 从"为什么"到"怎么做"

上一课我们用 Saga 完成了 Trigger Map——搞清楚了用户的心理驱动力和功能优先级。现在要回答下一个问题：

> "知道了用户想要什么，具体的用户旅程该怎么设计？"

这就是 UX 场景（Scenarios）的工作——它是连接"战略层"和"执行层"的桥梁。

```
Trigger Map（用户为什么来）
    ↓
UX 场景（用户走什么路径）← 本课内容
    ↓
概念规范（每个页面长什么样）
    ↓
设计系统 + 线框图（最终视觉呈现）
```

### 什么是 UX 场景？

UX 场景不是传统意义的"用户流程图"。WDS 对场景有一个很精确的定义：

> **场景 = 用户从进入产品到完成某个心理驱动力的完整路径（Sunshine Path）**

注意——场景关注的是"完成心理驱动力"，不是"完成一个功能操作"。

**例子**：

| 传统用户流程 | WDS UX 场景 |
|------------|-----------|
| "用户完成注册" | "用户从'好奇'到'信任'的旅程" |
| "用户下单购买" | "用户从'发现'到'满足'的旅程" |
| "用户提交反馈" | "用户从'困惑'到'被听到'的旅程" |

看到区别了吗？WDS 的场景描述的是**心理状态的转变**，而不仅仅是操作步骤。

### 场景大纲（Outline Scenarios）工作流

这是 WDS 的阶段 3，由 Freya 负责执行。整个工作流分为 9 个步骤：

```
Step 01: 加载上下文
↓ 读取产品简报、Trigger Map 等前序文档

Step 02: 分析范围（用户检查点）
↓ 确定网站类型、页面数量、规模策略

Step 03: 构建战略上下文
↓ 从 Trigger Map 中提取战略信息

Step 04: 建议场景方案（用户检查点）
↓ 提出场景计划，等你审批

Step 05: 详细描述每个场景
↓ 逐个场景展开，包括步骤、触发点、预期情绪

Step 06: 生成场景索引
↓ 创建 00-ux-scenarios.md 索引文件

Step 07: 质量审查
↓ 对照质量检查清单自检

Step 08: 更新设计日志
↓ 在 Design Log 中记录阶段 3 完成

Step 09: 交接
↓ 准备移交给阶段 4（概念规范）
```

**关键特性**：Step 02 和 Step 04 是"用户检查点"——Freya 会在这两个节点停下来等你确认，不会擅自往下走。这保证了 PM 对设计方向的控制权。

### 认识 Freya 的三种工作模式

Freya 和 Saga 一样，支持三种工作模式，但在 UX 场景阶段有不同的表现：

#### Discuss（讨论模式）

```
Freya：根据 Trigger Map，我觉得我们需要 5 个核心场景。
       第一个场景是"首次发现旅程"——覆盖 Lily 的 DF+1（发现感）。
       你觉得这个场景的起点应该是什么？搜索引擎？社交分享？还是直接打开 App？

你：    我觉得应该从社交分享开始，因为我们的目标用户主要从小红书来。

Freya：好的，那这个场景的路径就是：
       小红书看到分享 → 点击链接 → 着陆页 → 浏览推荐 → 发现心仪商品
       你觉得"浏览推荐"这一步应该是瀑布流还是策展式？
```

这种模式适合你有明确想法、想要深度参与设计决策的时候。

#### Suggest（建议模式）

```
Freya：基于 Trigger Map 分析，我建议以下 5 个核心场景：

       场景 1：首次发现旅程（覆盖 DF+1 发现感）
       路径：社交入口 → 着陆页 → 个性化推荐 → 商品详情 → 收藏/加购
       预期情绪弧：好奇 → 惊喜 → 兴奋 → 行动冲动

       场景 2：购买决策旅程（覆盖 DF-1 后悔焦虑）
       路径：加购 → 查看评价 → 对比 → 确认下单 → 安心支付
       预期情绪弧：犹豫 → 求证 → 确信 → 安心

       ...

       你要调整哪个场景？还是全部通过？
```

这种模式适合你信任 Freya 的专业判断，只想做审核和微调。

#### Dream（AI 自主模式）

```
Freya：我已经基于 Trigger Map 自动生成了 5 个核心场景的完整大纲。

       所有场景文件已保存到：
       - C-UX-Scenarios/01-discovery-journey.md
       - C-UX-Scenarios/02-purchase-decision.md
       - C-UX-Scenarios/03-social-sharing.md
       - C-UX-Scenarios/04-return-visit.md
       - C-UX-Scenarios/05-loyalty-upgrade.md

       每个场景包含：路径步骤、触发因素引用、预期情绪、
       页面列表、关键交互点。

       请审阅，如有修改意见请告知。
```

这种模式适合时间紧张、想快速推进的时候。

### 概念规范（Conceptual Specs）

场景大纲完成后，下一步是概念规范——这是 WDS 阶段 4 的工作。

概念规范是什么？简单说，就是**用文字精确描述每个页面的每个元素和每个状态**——比视觉稿更详细。

**举个例子**——注册页面的概念规范（节选）：

```markdown
## 注册页面概念规范

### 页面目标
- 触发因素引用：DF-1（后悔焦虑 → 降低门槛）
- 预期转化率：> 60%
- 目标停留时间：< 90 秒

### 元素清单

#### 1. 品牌 Logo
- 位置：顶部居中
- 尺寸：48x48px
- 目的：品牌识别，建立信任

#### 2. 欢迎语
- 内容："发现属于你的潮流"
- 字体：H2，品牌主色
- 目的：强化 DF+1（发现感），暗示个性化

#### 3. 社交登录按钮组
- 排列：微信 > 手机号 > 邮箱
- 微信排第一的原因：目标用户使用频率最高，降低注册摩擦
- 按钮样式：圆角，品牌色描边

#### 4. 信任标识
- 内容："已有 50,000+ 潮人加入"
- 位置：社交登录下方
- 目的：社会证明，消除 DF-1（后悔焦虑）
- 状态变化：数字每天自动更新

### 状态定义

| 状态 | 描述 | 触发条件 |
|------|------|---------|
| 默认 | 显示社交登录选项 | 页面加载 |
| 加载中 | 社交登录按钮变为加载动画 | 用户点击社交登录 |
| 错误 | 显示错误提示（红色，底部） | 登录失败 |
| 成功 | 跳转到首页 + 欢迎弹窗 | 登录成功 |
```

**PM 为什么要关心概念规范**：

1. **验收标准的精确版**：概念规范写清楚了"每个元素该长什么样、在什么情况下变成什么状态"——这就是最精确的验收标准
2. **设计和开发不打架**：有了概念规范，设计师不会"自由发挥"，开发也不会"理解偏差"
3. **每个元素都有理由**：注意看上面的规范——"微信排第一的原因"、"信任标识的目的"——每个设计决策都有追溯

### 实操：设计"用户注册"的完整 UX 场景

**场景定义**：用户从"好奇"到"信任"的注册旅程

**Trigger Map 引用**：
- 覆盖驱动力：DF-1（后悔焦虑 → 降低门槛）、DF+1（发现感 → 激发好奇）

**启动 Freya**：

```bash
/bmad-agent-wds-freya-ux
# 选择场景设计工作流
```

**场景路径设计**：

```
步骤 1：着陆页（情绪：好奇）
├── 用户从社交平台点击链接进入
├── 看到"发现属于你的潮流"
├── 浏览精选商品预览（不需要登录就能看）
└── 心理：被吸引，想看更多

步骤 2：触发注册（情绪：轻微犹豫）
├── 用户点击"收藏"或"加购"
├── 弹出注册提示："登录后即可收藏"
├── 提示文案强调简单："微信一键登录，3秒完成"
└── 心理：门槛很低，试试看

步骤 3：注册页（情绪：需要安全感）
├── 社交登录按钮（微信/手机号/邮箱）
├── 信任标识（"50,000+ 潮人已加入"）
├── 隐私保护提示（"我们不会发布任何内容"）
└── 心理：看起来安全，可以信任

步骤 4：兴趣选择（情绪：期待个性化）
├── 选择 3-5 个感兴趣的风格标签
├── 文案："帮我们更懂你的风格"
├── 可跳过但鼓励完成
└── 心理：它在用心了解我

步骤 5：个性化首页（情绪：惊喜 + 归属感）
├── 基于兴趣标签的个性化推荐
├── 欢迎弹窗："你的专属潮流空间已就绪"
├── 之前收藏的商品自动出现在收藏夹
└── 心理：这个 App 是为我准备的
```

**这个场景设计的亮点**：

1. **先体验后注册**：用户在注册前就能看到商品，降低了心理门槛
2. **在"心动时刻"触发注册**：不是进来就让注册，而是在用户想"收藏"的时候才要求——此时用户已经有了动力
3. **每一步都有情绪设计**：从"好奇"到"信任"到"惊喜"，是一个完整的情绪弧线
4. **每个文案都有心理学依据**："50,000+ 潮人"是社会证明，"3 秒完成"是降低感知成本

### 设计交付（Design Delivery）

完成场景和概念规范后，WDS 还有一个"设计交付"阶段——确保所有规范是完整的、没有遗漏的。

Freya 会做以下检查：

```
✅ 所有场景都有完整路径
✅ 所有页面都有概念规范
✅ 所有元素都有状态定义
✅ 所有交互都有说明
✅ 所有决策都记录在 Design Log 中
✅ 触发因素引用完整
✅ 无歧义，无遗漏
```

只有全部通过，才算"设计完成"——可以移交给开发团队。

## 常见问题

**Q: 概念规范太详细了，开发真的需要这么细吗？**

A: 需要。"概念规范太细"导致的返工成本是 0，"概念规范太粗"导致的返工成本是 10x。每一个"这里该怎么处理？"的问题都意味着一次沟通、一次等待、一次可能的误解。概念规范的目标就是让开发"不需要问任何人就能实现"。

**Q: 一个产品需要多少个场景？**

A: 取决于产品复杂度。Freya 会在 Step 02（分析范围）帮你评估。一般来说：小型产品 3-5 个核心场景，中型产品 8-12 个，大型产品 15-20 个。每个场景覆盖 1-3 个心理驱动力。

**Q: 场景设计和 BMM 的 Epics/Stories 怎么对应？**

A: 场景和 Epics 不是一对一的关系。一个场景可能跨越多个 Epic（比如"注册场景"涉及到"用户认证 Epic"和"个性化推荐 Epic"），一个 Epic 也可能被多个场景引用。但两者是互补的——场景关注"用户体验"，Epic 关注"开发交付"。在做 Story 拆分时，可以用场景来验证 Story 的用户价值。

**Q: 三种模式可以混着用吗？**

A: 完全可以。你可以在场景级别切换模式——比如核心场景用 Discuss 模式深入讨论，次要场景用 Dream 模式快速生成。Freya 支持随时切换。
