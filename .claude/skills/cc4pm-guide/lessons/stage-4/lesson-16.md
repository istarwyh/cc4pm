# Lesson 16: 质量三件套：/tdd + /e2e + /code-review

## 本课目标

- 理解 TDD（测试驱动开发）的 Red-Green-Refactor 循环
- 掌握 E2E（端到端测试）的核心概念和使用场景
- 理解 Code Review 的分级体系和自动化审查
- 学会从 PM 角度使用质量报告做发布决策

## 核心内容

### 为什么 PM 需要懂质量工具？

你可能觉得"测试和代码审查是开发的事"。没错，执行确实是开发的事——但**质量标准的制定和发布决策**是 PM 的事。

想象一下这个对话：

```
开发：代码写完了，可以上线了。
PM：测试通过了吗？
开发：通过了。
PM：覆盖率多少？
开发：呃……我不知道什么叫覆盖率。
PM：有安全审查吗？
开发：什么安全审查？
PM：（冷汗）
```

你不需要自己写测试，但你需要**能看懂质量报告、能定义质量标准、能做出"能不能上线"的判断**。

cc4pm 提供了三个质量工具，我们一个个来。

### 工具一：/tdd —— 测试驱动开发

**TDD 是什么？**

一句话：**先写测试，再写代码。**

传统开发流程：写代码 → 手动测试 → 发现 Bug → 修 Bug → 再测试...

TDD 流程：写测试 → 测试失败（RED）→ 写代码 → 测试通过（GREEN）→ 优化代码（REFACTOR）

```
RED → GREEN → REFACTOR → REPEAT

RED:      写一个会失败的测试（因为功能还没做）
GREEN:    写最少的代码让测试通过
REFACTOR: 优化代码结构，保持测试通过
REPEAT:   开始下一个功能
```

**/tdd 命令的使用**：

```bash
/tdd 实现积分抵扣功能，验收标准：
- 100积分抵扣1元
- 单笔最多抵扣50元
- 积分不足时按实际积分抵扣
- 抵扣后金额不能为负
```

tdd-guide 代理会自动：

1. 定义接口（输入输出的数据结构）
2. 写失败的测试（覆盖所有验收标准 + 边界情况）
3. 运行测试确认失败（确认测试是有效的）
4. 写最少的代码让测试通过
5. 运行测试确认通过
6. 优化代码
7. 检查覆盖率（目标 80%+，关键逻辑 100%）

**PM 为什么要关心 TDD**：

| TDD 的价值 | PM 能看到什么 |
|-----------|-------------|
| Bug 更少 | 上线后的 Bug 率降低 60-80% |
| 需求精确 | 测试就是需求的"可执行版本" |
| 安全修改 | 改代码后测试自动验证没有"改坏" |
| 覆盖率可量化 | "80% 覆盖率"比"我觉得没问题"靠谱 |

### 工具二：/e2e —— 端到端测试

**E2E 是什么？**

如果说 TDD 是检查"每个零件是否合格"，E2E 就是检查"整台机器能不能跑起来"。

E2E 模拟真实用户的完整操作流程——打开浏览器、点击按钮、输入文字、等待响应、检查结果。

**/e2e 命令的使用**：

```bash
/e2e 测试用户购买流程：
1. 用户登录
2. 搜索商品"运动鞋"
3. 选择第一个搜索结果
4. 加入购物车
5. 使用优惠券
6. 确认下单
7. 完成支付
```

e2e-runner 代理会自动：

1. 分析用户流程，识别测试场景
2. 生成 Playwright 测试代码（一个流行的自动化测试框架）
3. 在多个浏览器上运行测试（Chrome、Firefox、Safari）
4. 截图记录每一步的结果
5. 失败时自动录制视频和操作追踪
6. 生成测试报告

**E2E 测试报告示例**：

```
╔══════════════════════════════════════════════════╗
║              E2E Test Results                     ║
╠══════════════════════════════════════════════════╣
║ Status:     PASSED                                ║
║ Total:      7 tests                               ║
║ Passed:     7 (100%)                              ║
║ Failed:     0                                     ║
║ Flaky:      0                                     ║
║ Duration:   12.3s                                 ║
╚══════════════════════════════════════════════════╝
```

**PM 的 E2E 关注点**：

| 你关心的 | E2E 告诉你什么 |
|---------|-------------|
| 核心流程能走通吗 | 100% 通过 = 可以上线 |
| 哪些流程有风险 | 失败的测试 = 流程有 Bug |
| 多浏览器兼容 | Chrome/Firefox/Safari 分别的结果 |
| 性能是否达标 | 每步操作的响应时间 |

### 工具三：/code-review —— 自动化代码审查

**Code Review 是什么？**

代码审查就是"让别人检查你写的代码"。传统方式是同事互相审查——耗时、主观、容易遗漏。

cc4pm 的 `/code-review` 是**自动化审查**，而且审查标准比人严格得多。

**审查的四个维度**：

```
CRITICAL（致命）—— 必须修复，否则不能上线
├── 硬编码的密码、API Key
├── SQL 注入漏洞
├── XSS 跨站脚本漏洞
├── 缺少输入验证
└── 不安全的依赖

HIGH（严重）—— 必须修复
├── 函数超过 50 行
├── 文件超过 800 行
├── 嵌套超过 4 层
├── 缺少错误处理
└── 遗留的 console.log

MEDIUM（中等）—— 建议修复
├── 使用了可变数据模式
├── 新代码缺少测试
└── 可访问性问题

LOW（轻微）—— 可以后续优化
├── 命名不够清晰
└── 代码风格不一致
```

**使用方式**：

```bash
/code-review
```

系统会自动检查所有未提交的代码变更，生成报告。

**审查报告示例**：

```markdown
## Code Review Report

### Summary
- Files reviewed: 8
- Issues found: 3
- CRITICAL: 0
- HIGH: 1
- MEDIUM: 1
- LOW: 1

### HIGH Issues

1. [HIGH] src/services/points.ts:45
   Issue: Missing error handling for database transaction
   Suggested fix: Wrap in try-catch with rollback on failure

### MEDIUM Issues

2. [MEDIUM] src/api/checkout/route.ts:23
   Issue: New endpoint missing unit tests
   Suggested fix: Add test file with happy path + error cases

### LOW Issues

3. [LOW] src/utils/format.ts:12
   Issue: Function name 'calc' is not descriptive
   Suggested fix: Rename to 'calculatePointsDiscount'

### Recommendation: NEEDS WORK
Fix HIGH issue before merging.
```

### 质量三件套的配合使用

三个工具不是孤立的，它们有一个最佳使用顺序：

```
/plan        ← 先做计划
↓
/tdd         ← 测试驱动开发（写代码时）
↓
/code-review ← 代码审查（代码写完后）
↓
/e2e         ← 端到端测试（功能完成后）
↓
质量门禁     ← 发布前最终检查
```

### PM 的质量决策框架

作为 PM，你需要做的"质量决策"本质上就一个：**能不能上线？**

**决策标准**：

| 维度 | 通过标准 | 使用工具 |
|------|---------|---------|
| CRITICAL 问题 | 0 个 | /code-review |
| HIGH 问题 | 修复后 0 个 | /code-review |
| 测试覆盖率 | > 80% | /tdd |
| 核心流程测试 | 100% 通过 | /e2e |
| 安全审查 | 无高危漏洞 | /code-review --security |

**快速判断口诀**：

```
CRITICAL = 0：不能有致命安全问题
HIGH = 修复：必须修复，可以有修复记录
80% 覆盖率：核心逻辑有测试保护
E2E 100%：用户核心流程走得通
```

满足这四条，可以安全上线。不满足任何一条，就需要继续修复。

### Security Review：安全漏洞检测

除了一般的 code-review，cc4pm 还有专门的安全审查代理。重点检查：

- 硬编码的凭据（密码、密钥、Token）
- SQL 注入风险
- XSS（跨站脚本）风险
- 路径遍历风险
- 不安全的依赖包

**PM 需要知道的安全常识**：

```
如果安全审查报告中出现以下任何一项，立即暂停上线：

CRITICAL: Hardcoded API key found  → 密钥泄露
CRITICAL: SQL injection risk       → 数据库可能被攻击
CRITICAL: Missing authentication   → 接口没有鉴权
CRITICAL: Sensitive data exposure   → 用户数据可能泄露
```

这些不是"后续优化"的问题，是"现在不修、明天上新闻"的问题。

## 常见问题

**Q: 作为 PM，我需要能运行这些命令吗？**

A: 理想情况下能。但即使你不直接运行，你也需要：(1) 知道应该在什么时候使用哪个工具；(2) 能看懂输出报告；(3) 能基于报告做发布决策。最低限度——你应该能看懂 code-review 报告和 E2E 测试报告。

**Q: 80% 覆盖率是怎么来的？**

A: 这是行业共识的平衡点。低于 80%，太多代码没有测试保护，改动容易引入 Bug。高于 95%，投入产出比急剧下降（剩下的 5% 可能需要花 50% 的精力去覆盖）。但对关键业务逻辑（比如支付、积分计算），目标应该是 100%。

**Q: E2E 测试和 QA 手工测试有什么区别？**

A: E2E 是自动化的——写一次，可以反复运行。QA 手工测试是人工的——每次都需要人去操作。E2E 适合"确定性高、路径固定"的核心流程；QA 手工测试适合"探索性的、需要直觉判断"的场景。两者互补，不是替代关系。

**Q: 代码审查报告里的 MEDIUM 和 LOW 问题可以忽略吗？**

A: 短期可以。MEDIUM 和 LOW 问题不会阻塞上线，但长期积累会变成"技术债"。建议的策略：CRITICAL 和 HIGH 立即修复，MEDIUM 排入下一个 Sprint，LOW 记录到技术债清单中定期清理。
