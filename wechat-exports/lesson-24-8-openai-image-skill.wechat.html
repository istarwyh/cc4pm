CC4PM LESSON 24.8

把文章变成素材包：封面、配图、PPT 一次落到本地

生图能力放进 Claude Code 后，产物开始有路径、命名、记录和复用方式。

你今天写公众号，明天要改成飞书方案，后天又要拿去做路演 PPT。每次临时打开网页生一张图，当然能解决眼前问题。麻烦在后面：那张图的提示词在哪，尺寸为什么是这个，能不能再出三张同风格配图，下周谁还能复现。

产品主理人缺的往往不是“画图入口”。真正缺的是一套会留下痕迹的素材工程。输入是一篇文章、一个产品想法、一组用户数据，输出不只是 PNG，还包括提示词、模型、尺寸、文件名、版本和用途。

先跑通，再理解

如果你现在只想先拿到一张图，在 Claude Code 里直接把下面这段话发给它。不要自己逐行敲命令，让 Claude Code 帮你检查、安装、刷新和验证。

直接发给 Claude Code

请帮我安装 openai-plugin。
添加 istarwyh/agent-plugins 插件市场，安装 openai-plugin@agent-plugins。
安装后确认 openai-plugin 已启用；如果 Skill 没出现，请执行 /reload-plugins。
然后检查 codex CLI 和登录状态，用 codex-image 生成一张暖色系公众号封面。
保存到 generated/openai-images，生成后告诉我文件路径，并用 file 验证是不是 PNG。

openai-plugin 安装好以后，codex-image 和 image-skill 都来自这个插件，不需要分开装。一个负责默认的 Codex 登录路线，一个负责已有 provider 的接口路线。

如果你要自己检查，实际命令大概是这些。读者不需要背，知道它们存在就行。

claude plugin marketplace add istarwyh/agent-plugins
claude plugin install openai-plugin@agent-plugins --scope user
claude plugin list
codex login status

图片一旦进入工作流，就不再只是“好不好看”的结果。它会变成一份可以复盘、比较、重做的产品资产。

旧做法很短：prompt → image。它适合一次性创作，也适合试手感。问题是，产品工作从来不是只要一张图。你要封面、正文配图、PPT 页图、社媒图、活动海报，还要尽量像一个品牌做出来的。

课程里把这条链路写得更完整：content / code / data / idea → visual plan → image variants → evaluation → selection → remix → document / deck / UI / test dataset / brand assets。翻成日常工作，就是先拆用途，再出多版，再挑图，再改成封面、PPT、长图或测试素材。

这也是把生图做成 Skill 的意义。Claude Code 不是把你一句话原封不动丢给图片模型。它可以先读你的文章，判断受众，拆出每张图的用途，给出统一风格，再把每次生成的文件落到项目目录里。

这里要留一点边界。模拟图片适合做 OCR、视觉检查、排版压力测试，不能替代真实业务数据验证。它帮你补足“场景长相”，不替你证明业务规则一定正确。

装好后默认先用 codex-image

课程推荐的默认路线是 codex-image。它复用 Codex 自己的 OAuth 登录态，让 Codex CLI 去调用内置 image_gen。Claude Code 负责理解需求、抽取尺寸和数量、指定输出目录、复制文件、检查格式。

你在 Claude Code 里可以这样说：

帮我用 codex-image 生成一张暖色系产品发布封面，
保存到 generated/openai-images，尺寸 1536x1024，质量 high。

背后的链路大概是这样：Claude Code 触发 openai-plugin 里的 codex-image；Skill 调用 codex exec；Codex CLI 用自己的登录态生成图片；图片先落在 ~/.codex/generated_images/，再复制到你指定的项目目录。

这里有个坑很容易踩：Codex OAuth token 不能拿去当 OpenAI REST API Key。它不是给 /v1/images/generations 用的。你只需要让 Codex CLI 自己处理认证，不要把 token 抠出来再塞进别的请求里。

有 provider 时再用 image-skill

image-skill 走的是 OpenAI-compatible 路线。它用本地 openai CLI 请求 /v1/images/generations，拿到 b64_json 后直接解码成本地 PNG。如果你公司或团队已经给过 OpenAI 兼容接口，就用这个；没有就先别碰，直接从 codex-image 开始。

用它时可以这样说：

我已经有 OpenAI-compatible provider。
请用 image-skill 生成一张 1024x1024 的公众号封面，保存到 generated/openai-images。
如果没有 provider memory，请先检查 OPENAI_BASE_URL、OPENAI_API_KEY、OPENAI_IMAGE_MODEL。
生成后告诉我文件路径，并用 file 验证 PNG。

这段最要紧的地方不在命令本身。别把图片结果打印到对话里。base64 很长，会污染上下文，也不方便审查。正确做法是让命令直接抽取字段、解码、写文件，然后用 file 校验 PNG。

本地素材包长这样

visual-assets/product-launch/
├── source-article.md
├── visual-plan.json
├── prompts/
│   ├── 01-cover.md
│   ├── 02-flow-diagram.md
│   └── 03-deck-slide.md
├── images/
│   ├── cover-20260531-101200.png
│   ├── article-01-20260531-101245.png
│   └── slide-01-20260531-101330.png
└── manifest.md

manifest.md 里只要记住四项就够了：Prompt、Model、Size、Saved file。下次你要改封面，不必从零开始猜提示词；你能直接打开旧记录，知道哪种描述稳定，哪个尺寸适合公众号，哪张图后来被团队采用。

Provider Memory 解决的是重复配置

image-skill 第一次跑通后，可以把 provider、base_url、model、size 记到本机。下次你只说“生成一张封面”，Claude Code 就知道继续沿用那套配置。

但真实第三方 API Key 不应该明文写进磁盘文件。更稳妥的写法是只保存环境变量名。这样文章、截图、仓库里都不会出现真实凭据。

{
  "provider": "custom-openai-compatible",
  "api_key_env": "OPENAI_API_KEY",
  "base_url_env": "OPENAI_BASE_URL",
  "model": "image-model-name",
  "size": "1024x1024"
}

如果你走的是 codex-image，这份 Provider Memory 通常用不上。认证状态交给 Codex CLI 管，Skill 只负责把产物放好。

Deck 是更容易感知的产物

单张图很容易被当成装饰。Deck 不一样。你把一篇产品分析文章交给 any2deck，它会先拆大纲，再给每页写提示词，随后逐页生成 PNG，最后合成 PPTX 和 PDF。建议让它输出到 slide-deck/{topic-slug}/，这样大纲、逐页提示词、PNG、PPTX、PDF 都在同一个目录里。

如果你的环境还没有 any2deck，把这句话发给 Claude Code，让它安装并验证：

请帮我安装 any2deck：执行 npx lovstudio skills add any2deck -g -y，
安装后确认 any2deck 可用，再告诉我如何把文章导出到 slide-deck/{topic-slug}/。

这个例子很适合理解生图在 Claude Code 里的位置。它已经离开了孤立的“画一张好看的图”，进入内容交付工序：先组织叙事，再拆成页面，再按页生成视觉，最后交付可编辑文件。

可以直接发给 Claude Code 的话

请把 product-analysis.md 做成 10 页中文 bold-editorial 风格 Deck。
先生成 outline.md 和 prompts/，让我确认叙事结构。
确认后逐页生成 PNG，并导出 PPTX 和 PDF。
每页记录 Prompt、Model、Size、Saved file。

最后再把安全边界说清楚。不要粘贴 b64_json，不要把真实 API Key 写进课程、仓库或截图，不要覆盖旧图。所有图片用时间戳命名，生成后用 file 验证格式。听起来有点啰嗦，但这正是“工作流”和“随手试一下”的区别。

下次你再做一篇发布文章，不必从“帮我画张封面”开始。你可以从素材包开始：这篇文章需要哪些图，每张图服务哪一段，哪些风格必须一致，产物落到哪里，之后怎么复用。

AI 生成得越快，越要把产物放进可追踪的文件系统里。否则速度只会制造更多找不回来的临时结果。
