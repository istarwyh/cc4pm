 Harness Engineering：从理论到实战 —— 打造自我进化的 AI 软件系统

  引言：工程师角色的历史性转折

  在过去的一年中，大语言模型（LLM）的编码能力迎来了爆发式增长。从简
  单的函数补全到单文件的重构，AI
  已经展现出令人惊叹的单点突破能力。然而，当我们将这些强大的模型投
  入到真实的、具有长生命周期的复杂商业项目中时，却普遍遭遇了一堵“隐
  形的墙”。

  我们发现：AI
  在最初的几次交互中表现优异，但随着任务链条的拉长、上下文的增加，
  它开始出现“幻觉”、忘记最初的约束、为了修复一个 Bug 而制造两个新的
  Bug，甚至在上下文接近打满时开始敷衍了事、草草收工。

  这暴露出一个核心问题：大模型本身只是一个“函数”，它缺乏维持复杂工
  程系统长期稳定收敛的外部环境约束。

  正如 OpenAI 在大规模应用 Codex 智能体时所总结的：在 AI First
  的时代，工程师的核心角色正在发生根本性转变——从“编写功能代码”转向“
  设计环境、明确意图和构建反馈回路”。这个全新的工程学分支，我们称之
  为 Harness Engineering（驾驭层工程）。

  本文将系统性地阐述 Harness Engineering
  的核心理论，并结合我们在千万级数据量项目（HelixVerify）中的真实落
  地案例，展示如何通过构建端到端的自举（Bootstrapping）闭环，让 AI
  实现 7×24 小时的无人值守自主进化。

  ---

  第一部分：理论基石 —— 为什么 AI 需要被“驾驭”？

  要理解
  Harness，首先必须深刻理解大模型在执行长程任务时的三种典型失效模式
  。这三种模式决定了我们不能简单地将一个需求直接扔给 AI
  并期望它自动完成。

  1.1 作者偏见：不能让 AI 既当“运动员”又当“裁判”
  在早期的单体 Agent 设计中，开发者通常会让 AI
  编写完代码后，再附加一句提示词：“请检查你的代码是否符合要求”。
  Anthropic 的工程团队通过大量实验发现了一个致命的规律：AI
  在评估自己的产出时，会产生系统性的虚高打分。
  这与人类的心理学现象如出一辙——AI
  会倾向于为自己生成的逻辑漏洞寻找合理化解释。

  破局点：生成器-评估器分离 (Generator-Evaluator Split)。 Harness
  的第一原则是在架构层面强制分离执行者和审查者。生成代码的 AI
  进程（Generator）与运行审查逻辑的 AI
  进程（Evaluator）必须拥有完全隔离的上下文。评估器从未参与过代码的
  创作过程，因此能保持绝对的“怀疑主义”和客观性。

  1.2 上下文焦虑：AI 的“草草收工”综合症
  当一个 Agent 在同一个会话（Session）中持续运行，随着消耗的 Token
  逐渐逼近窗口上限，模型会产生一种被称为“上下文焦虑” (Context
  Anxiety) 的行为变异。
  具体表现为：原本应该详细实现的模块被简化为 // TODO，原本该写的 10
  个测试用例缩减为 3
  个，并主动建议“剩余的部分我们下次再做”。它并非真的没有空间了，而
  是“感觉”空间不够了。

  破局点：重置而非压缩 (Reset vs. Compaction)。 传统的做法是让 AI
  总结之前的对话来压缩上下文，但这无法消除它的焦虑。Harness
  提倡使用“接力跑”模式——当一个微小里程碑完成后，彻底销毁当前 AI
  进程，清空上下文，通过标准化的文件（如
  SHARED_TASK_NOTES.md）将状态传递给下一个全新启动的、100% 清醒的
  AI 进程。

  1.3 目标偏移：没有终点线的马拉松
  如果一个任务连“什么算完成”都无法被结构化地定义，将其交给自主运行
  的 AI 是极其危险的。AI
  可能会无止境地发散，或者在偏离业务意图的奇技淫巧上耗费大量算力。

  破局点：Sprint 合约 (Sprint Contracts)。 在 Harness
  的闭环中，Generator 在开始写代码前，必须先与 Evaluator
  对齐“验收标准”。这个标准不是限定它具体怎么写，而是规定“完成后必须
  满足的客观条件”。先签合约，再动代码，这是遏制发散的唯一手段。

  第二部分：量化引擎 —— EDD (评估驱动开发)

  如果 Harness 是一台发动机，那么评估驱动开发 (Eval-Driven
  Development, EDD) 就是它的燃料。Harness
  的“闭环优化”不能建立在人类的肉眼观察上，必须建立在严谨的数学度量
  上。

  2.1 为什么 TDD 在 AI 时代不够用了？
  传统软件工程基于 TDD（测试驱动开发），测试用例要么过
  (Pass)，要么不过 (Fail)。但 AI 具有概率性，同一个 Prompt
  今天成功，明天可能因为模型底座微调或温度（Temperature）采样而失败
  。

  在 AI 时代，主观感受必须让位于客观数字。我们需要
  EDD：先定义量化的评估集（Benchmark），再让 AI
  迭代，最后用统计学指标来衡量可靠性。

  2.2 核心指标：Pass@k 与 Pass^k
  为了在 Harness 闭环中衡量 AI 的稳定性，我们引入了以下度量标准：
   * Pass@1：一次请求就成功的概率。这是衡量成本和效率的核心指标。Pa
     ss@1 极低意味着极高的 Token 浪费。
   * Pass@3：重试 3
     次内至少成功一次的概率。这是衡量方案“可行性”的指标。
   * Pass^3：连续 3
     次请求全部成功的概率。这是衡量“回归稳定性”的指标，核心业务链路
     必须追求 Pass^3 接近 100%。

  2.3 构建两类评估器 (Graders)
  在 Harness 的“评估”节点中，我们通常挂载两类引擎：
   1. Code Grader (代码评估器)：完全确定性的脚本。例如使用 grep
      检查是否输出了特定关键字，使用 npm run test
      检查单元测试是否通过。
   2. Model Grader
      (模型评估器)：针对主观或语义判断。挂载另一个强大的
      LLM，为其提供极其详尽的评分细则（Rubric），让其对产出进行 1-5
      分的打分。

  只有当这套 EDD 量化体系就绪，Harness 才能进行无人值守的闭环迭代。
  不过单纯的 EDD 是不够的——它能告诉你“分数掉了”，但它不能告诉你“怎么提分”。所以必须将评分和评分规则也给到 AI。

  第三部分：实战演练 —— HelixVerify 项目的 114 次自举迭代

  理论再完美，也需要实战的检验。

  3.1 业务背景与困境
  HelixVerify 是我们在“保险查查”业务中开发的一个 Agent
  产出质量验证系统。它的核心使命是通过后置验证，消除 LLM
  在生成保险详情页面时的幻觉与低质量内容。

  起初，我们面临着惨淡的基线数据：风险样本的召回率仅为
  8%，无风险样本的通过率仅为 36.11%。
  在早期的优化尝试中，我们采用的是传统的人工干预模式：人类分析错误
  日志（Badcase） -> 想到一个规则 -> 让 AI 写代码 ->
  人类运行评测看结果 -> 再次修改。
  这种模式下，迭代一次需要 5-7
  天，且成功率极低。随着规则的增多，系统变得脆弱不堪。

  3.2 破局：架构师指明方向，混合引擎打底
  我们意识到，单靠 Opus-4.5
  这样的顶级模型自己是无法迭代出一个优秀的系统架构的，它只会在微观
  逻辑上钻牛角尖。
  因此，我们作为人类“架构师”，做出了本项目中最重要的顶层设计干预：
  提出“Rule 引擎 + LLM 多轮检测”的混合验证架构。

   * Rule 引擎 (23条规则)：处理确定性错误（如 HTML 注释残留、JSON
     空数据），毫秒级拦截 30-40% 的明显错误。
   * LLM 多轮语义检测：并行 7
     个维度处理逻辑矛盾、数值不准确等需要深层推理的隐性错误。

  两者互补，确立了系统的骨架。但这仅仅是开始。

  3.3 搭建自举闭环 (Bootstrapping) 的五大模块
  为了让 AI 能够自主把召回率从 8% 提升到 98%
  以上，我们并没有自己去写每一条过滤规则，而是指导 AI
  逐步写出了“测试它自己”的代码环境。
  在这个“自举”过程中，每搭建好一个模块，AI 减少一项对人的依赖。

  模块一：端到端测试体系 (E2E) —— 赋予 AI“看懂”系统的能力
  AI 只能修改代码，但它不知道改完后系统链路里发生了什么。我们指导
  AI 搭建了全链路的 Trace 系统和 E2E 脚本。
  这让 AI
  能够像人类高级工程师一样，不仅看到最终抛出的错误，还能从海量日志
  中顺藤摸瓜，看懂 LLM
  在中间哪一步的“思考”出了偏差。这是闭环的起点。

  模块二：标准化评测 (verify_bench) —— 赋予 AI“量化”的尺子
  为了落实 EDD 理念，我们构建了
  verify_bench。它在每次代码变更后，强制生成两份机器可读的文件：
   1. metric.txt：结构化的数据，包含当前的 TP、FP、FN 及 Recall。AI
      用它来判断“我的代码合不合格”。
   2. detail.txt：逐条对比明细。将原始日志、人类标定的 Ground
      Truth、当前测试结果横向对齐。AI 用它来寻找“我为什么错”。
  至此，人类彻底退出了“肉眼审查结果”的环节。

  模块三：Badcase 自动诊断 —— 消除“人脑分析”的瓶颈
  过去，人工聚类 Badcase 是最耗时的。我们利用 Harness
  分离原则，引入了一个专门用于“诊断”的独立 Agent。
  它读取
  detail.txt，利用大模型强大的归纳能力，自动将几百个误报样本分类，
  并基于此自动生成候选的优化策略代码，甚至根据预估收益进行优先级排
  序。

  模块四：TP 对抗验证 —— 根治“按下葫芦起了瓢”
  在 AI 自主迭代中，我们遭遇了惨痛的教训：在第 105 次迭代时，AI
  为了减少误报，生成了一套极其严苛的规则，结果直接把真正的风险样本
  也拦截了，导致召回率从 90% 暴跌至 40%。
  为了保证改动的安全，我们引入了 TP (True Positive)
  对抗验证。作为一道质量门禁，任何过滤规则在 Commit
  之前，必须在正确样本集上跑通，确保不会“误杀”。
  实施这一防护网后，后期的 30
  次无人值守迭代，无一次发生召回率倒退。

  模块五：阶梯式验证 —— 极致的试错成本压缩
  全量 202 个样本的评测需要耗时 60
  分钟。如果每次微调都跑全量，一天最多迭代十几次。
  由于 AI 提出的策略有 70%
  最终是被废弃的，我们设计了“三分钟快速失败”的阶梯验证法。前置一个
  小样本验证子集，90% 导致明显劣化的逻辑在 3 分钟内即可被拦截。这让
  AI 的日试错次数提升了一个数量级。

  3.4 Loop 基础设施：实现 7×24 小时飞轮
  当上述 5 个模块就绪，我们引入了 Ralph-Loop 结合 Supervisor
  (主管代理)。

   * Ralph-Loop
     强制核对预设的“完成承诺”。如果没有达到指标，系统会保存当前实验
     结果，彻底销毁进程（避免上下文焦虑），然后启动新进程重新尝试。
   * Supervisor 则在更高维度进行周期性审计。它防范 AI
     陷入无限重复试错的死循环，并在遇到涉及业务价值判断的“十字路口”
     时，挂起任务，等待人类架构师的审批。

  3.5 最终战报：无可争议的工程胜利
  经过 114
  次迭代，其中没有一行业务逻辑代码由人类手写。我们不仅收获了卓越的
  业务指标：

   * 召回率 (Recall)：基线 87.42% 飙升至最终的 98.86%（提升
     11.44pp）。
   * 通过率 (Pass Rate)：基线 36.11% 提升至 54.93%（提升
     18.82pp）。
   * 漏报数 (FN)：从 19 个硬性锐减至仅仅 1 个（下降 94.7%）。

  我们更收获了惊人的工程效率：人工主导的单轮迭代周期从 5-7
  天，被压缩到了 Harness 闭环下的 3-4 小时。
  我们证明了：只要反馈环境搭建得足够完备，AI
  就能够通过自举实现复杂系统的跨越式进化。

  第四部分：架构演进 —— Agent 系统的终极形态

  HelixVerify
  展示了在项目级实现闭环的惊人威力。然而，当我们将视角拉高，面向企
  业级、超大规模的 Agent 矩阵时，传统的 Harness
  架构将面临底层的重构：走向“大脑、双手与会话的彻底解耦”（三权分立
  ）。

  4.1 耦合架构的痛点：带着枷锁跳舞
  在早期的 Agent
  中，大模型（大脑）、执行沙箱（双手）和对话记录（会话）是强绑定的“
  宠物 (Pets)”。
  这带来了致命的脆弱性：沙箱里的进程崩溃，对话的上下文就随之丢失；
  沙箱包含敏感的 API
  Key，一旦遭受提示词注入，整个系统权限瞬间沦陷。

  4.2 终极架构：抽象与解耦
  未来的企业级 Agent 架构，必然走向如下形态：

   1. 大脑 (Brain)：无状态的推理引擎。 模型与 Harness
      的决策逻辑被抽取出来，像“牛群
      (Cattle)”一样随时可被替换。如果一个推理进程卡死，瞬间拉起一个
      新的。
   2. 会话 (Session)：外部持久化对象。
      摒弃依赖模型窗口记忆的做法。所有的操作、决策全部写入外部的
      Session Log。大脑需要回忆时，通过 API
      精准检索知识切片。这彻底根除了“上下文焦虑”，也为跨越数月的超
      长程任务提供了可能。
   3. 双手 (Hands)：延迟置备的安全沙箱。
      只有当大脑决定要执行特定代码时，系统才动态分配沙箱。真实凭证
      （Token）锁在云端保险库，沙箱代码只能通过 Proxy
      中转。这种解耦同时将系统的首 Token 延迟 (TTFT) 降低了近 90%。

  这种演进证明了：Harness Engineering
  的空间并未随模型底座变强而消失，它只是向着更深层的系统架构和分布
  式协作迁移。

  ---

  结语：Human Steers. Agents Execute.

  回顾从理论到实战的整个旅程，Harness Engineering
  向我们揭示了一个不可逆转的技术趋势：

  在后大模型时代，业务的护城河不再是有多少熟练的“代码打字员”，也不
  再是花哨的单点 Prompt 技巧。真正的护城河，是能够建立起一套包含
  E2E 追踪、EDD
  评估、自动化诊断、对抗验证与解耦架构的自我进化环境。

  当我们把这套环境（Harness）搭建完毕后，人类就不再需要去修补每一处
  逻辑的裂缝。我们将重新站到“掌舵者”的位置，负责制定商业规则、定义
  价值验收标准（EDD）、指明系统架构方向；而
  AI，则在这个严密编织的反馈网络中，不知疲倦地执行、试错、修复，直
  至达到我们在物理世界中设定的那个满分。

  搭好环境，制定规则，剩下的，交由智能体去完成。 这就是 Harness
  Engineering 赋予现代工程团队的最高杠杆。
